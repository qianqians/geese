# Geese 引擎功能差距分析与可用引擎路线图

---

## A. 已完成且可直接使用的功能（成熟度 A/A-）

| 模块 | 评级 | 核心能力 |
|------|:----:|----------|
| **Forward+ 渲染管线** | A | Cluster 光照剔除、实例化渲染、CSM 级联阴影、IBL 三模式、ACES Tonemap、TAA、骨骼蒙皮 5 模式、渲染图 DAG、Shader Graph |
| **物理系统** | A | rapier3d 全面封装：多场景世界、刚体/碰撞体/关节、射线检测/碰撞事件、胶囊体控制器、远程物理 TCP、统一管理层、PyO3 绑定 |
| **相机系统** | A | FPS/Orbit/TopDown 三模式、视锥裁剪、平滑跟随/朝向 API、测试充分 |
| **事件总线** | A | 类型安全 RwLock 设计、批量分发、EcsBridge、测试覆盖（注：报告 C1 的 unsafe Sync 问题已被修复） |
| **状态同步** | A | Source Engine 风格插值、SnapshotBuffer、Lagged/Extrapolated、速度外推 |
| **AOI 兴趣管理** | A | 九宫格分桶、Enter/Leave 增量事件 |
| **配置系统** | A | TOML 驱动、serde 序列化、层级合并、已桥接到运行时 |
| **服务端三节点** | A- | Hub/Gate/DBProxy、多协议、RPC、服务发现、实体迁移（无 unwrap/panic 风险） |
| **场景图核心** | A- | Scene/SceneObject/Octree/Prefab/glTF 导入/动画状态机/序列化/Python 脚本组件 |
| **输入系统** | A- | 键盘/鼠标/手柄全覆盖、ActionMap、WinitBackend + GilrsBackend |

## B. 部分实现需完善的功能（有骨架但关键部分缺失）

| 模块 | 评级 | 已有 | 缺失 |
|------|:----:|------|------|
| **Deferred+ 管线** | C+ | G-Buffer + Cluster Culling + Lighting | 完全缺少阴影集成（无 shadow_pass/shadow_atlas/CsmUniform），shader 无 shadow map 采样 |
| **后处理链** | C | Bloom GPU pass、ACES Tonemap、TAA、枚举定义 | SSAO/SSR/DoF/MotionBlur 无 GPU shader 实现（静默跳过）；Bloom 双重 intensity bug；tonemap pipeline 颜色格式不匹配；PostProcessPipeline::process() 无调用者 |
| **音频系统** | B | 多通道混音、rodio 后端、反距离衰减模型 | 真正的空间化（binaural panning/HRTF）、多普勒效应、环境混响 |
| **地形系统** | B- | 高度图/LOD/流式加载/geo-clipmap、GPU 渲染器 | 纹理 splatting、细节层(grass)、碰撞体生成 |
| **VFX 粒子** | B- | CPU 粒子模拟、多种发射器、Billboard 模式 | GPU 粒子模拟、力场、碰撞、子发射器、trail/ribbon |
| **ECS** | C+ | 基本 CRUD、EcsBridge | String 实体 ID（非 generational index）、O(n) 查询、无 System 调度/并行、无 Query 迭代器 |
| **NavMesh** | C+ | A* 寻路 + 漏斗平滑 | Recast/Detour 体素生成、动态障碍物、Off-mesh connection |
| **角色动画** | B | 状态机 + BlendTree + glTF 采样 + Marker 事件 | IK、动画层(layered blending)、additive blending、动画重定向 |
| **游戏运行时** | B- | winit+wgpu 循环、Android 入口、配置桥接 | iOS/WASM 实际接入有限、音频/输入未集成进主循环 |
| **Inspector 面板** | C | Transform 显示、材质编辑 | sync_transform() 为空函数（缓存未命中回退 [0,0,0]）、Mesh Renderer 显示占位文本 |
| **游戏内 UI** | D | `crates/ui/` 基于 egui 的通用封装 | 无专用 HUD/Widget 系统、无富文本、无动态 UI 布局 |
| **Launcher 模板** | C | 选择模板→生成工程流程 | 生成代码为空壳 |

## C. 完全缺失的关键功能（可用引擎必须具备）

| 缺失功能 | 重要度 | 说明 |
|----------|:------:|------|
| **Sprite / 2D 渲染** | P0 | 全项目无任何 2D 渲染支持，大多数游戏类型都需要 |
| **游戏内 UI 系统** | P0 | 无 HUD、血条、菜单、按钮、文本渲染（非编辑器 UI） |
| **光照贴图烘焙** | P1 | 仅有 IBL 环境光照，无传统 Lightmap UV 烘焙 |
| **反射探针** | P1 | 无反射探针系统，影响室内/金属场景的真实感 |
| **体积雾/云** | P1 | 无任何雾效实现，影响氛围渲染 |
| **水面/流体渲染** | P1 | 无水面渲染，影响自然环境场景 |
| **贴花系统 (Decal)** | P1 | 无 Decal 支持，影响弹痕/血迹/标识等常见效果 |
| **IK 逆向运动学** | P1 | 动画系统缺少 IK，影响角色脚部适配/瞄准等 |
| **存档系统** | P1 | 无游戏存档/读档机制 |
| **本地化/国际化** | P2 | 无 i18n 支持，影响多语言发行 |
| **触屏输入** | P2 | InputEvent 无 Touch/Pointer 事件，移动端受限 |
| **回放/录制系统** | P2 | 无游戏回放功能 |
| **AI 行为树（原生）** | P2 | 仅有 Python 侧 behavior3py，Rust 核心无原生行为树 |
| **粒子编辑器** | P2 | 无可视化粒子效果编辑器 |

## F. 与成熟引擎对比总览

| 能力维度 | Unity/Unreal | Godot | Geese 现状 | 差距 |
|----------|:----------:|:-----:|:----------:|------|
| 3D 渲染 | 完整 | 完整 | Forward+ 完整，Deferred+ 缺阴影 | 中 |
| 后处理 | 10+ 效果 | 8+ 效果 | Bloom+TAA+ACES，缺 4 种 | 中 |
| 2D 渲染 | 完整 | 完整 | 不存在 | 大 |
| 物理 | 完整 | 完整 | rapier3d 完整 | 小 |
| 动画 | IK+蒙太奇+蓝图 | IK+BlendTree | 状态机+BlendTree，缺 IK | 中 |
| 音频 | 完整 3D | 完整 3D | 基础框架+衰减，缺空间化 | 中 |
| 脚本 | C#/Lua | GDScript/C# | Python (PyO3) | 小 |
| UI | 完整 UGUI/UMG | 完整 Control | 仅 egui 封装 | 大 |
| 地形 | 完整 | 基础 | LOD+Streamer，缺纹理层 | 中 |
| 网络 | Netcode/Mirror | 基础 | 三节点完整 | 小 |
| 多平台 | 全平台 | 全平台 | 桌面+Android，iOS/WASM 有限 | 大 |
| 编辑器 | 完整 | 完整 | 10+ 面板，Inspector 需完善 | 中 |

**结论**：Geese 引擎在 3D 渲染（Forward+）、物理、网络三大支柱已具备生产级能力。走向"可用游戏引擎"的最大缺口是：**游戏内 UI 系统**、**2D 渲染**、**Deferred+ 阴影**、**后处理效果链**和**多种视觉效果（雾/水/反射探针/贴花）**。建议按 P0→P1→P2 顺序推进，预计 P0+P1 完成后可达到"基本可用的 3D 游戏引擎"水平。
